1. Why this exists

Converting between DOCX and PDF the normal way — opening an app, navigating Export, choosing a destination, waiting for the app to launch — adds up fast when done a dozen times a day. This tool collapses that to a single right-click with no app windows, no dialogs, and output that lands next to the original file.

2. DOCX → PDF: the long road

First attempt: AppleScript / JXA driving Microsoft Word

Bug Word 16.109.1 regression — all PDF export verbs return -1708

Word 16.109.1 (released late 2024) silently broke every AppleScript and JXA verb that previously worked for PDF export: saveAs, exportAsFixedFormat, and do Visual Basic all return -1708 errAEEventNotHandled. The do Visual Basic approach also produces a syntax error (-2741) because the command isn't in the dictionary at all in this version.

This ruled out the docx2pdf Python library (which drives Word internally), any direct Word automation, and any approach requiring Word to be installed.

Second attempt: textutil + Chrome headless

Trade-off Always available, but loses formatting

macOS ships textutil, which converts DOCX to HTML. Chrome (or Chromium) can then render that HTML to PDF via --headless=new --print-to-pdf.

Problem: textutil strips most Word formatting — tables become flat text, complex layouts collapse, fonts revert to system defaults. The resulting PDFs were described as "garbage" during testing. This path is kept as a last-resort fallback only.

Final approach: LibreOffice headless (primary)

Decision LibreOffice headless is the right default

soffice --headless --convert-to pdf --outdir <dir> <input> converts DOCX to PDF with high fidelity — fonts, tables, layouts, track changes, and complex structures all survive. The process is fully silent (no windows, no Dock icon) and takes ~1–3s per document.

The only downside is installation size (~500 MB). The installer checks for LibreOffice and warns if it's absent, falling through to the textutil+Chrome path rather than failing hard. 

Priority order:
  1. LibreOffice headless  → high quality, silent, no Word needed
  2. textutil + Chrome     → low quality, always available, Chrome must be installed

3. PDF → DOCX

Why pdf2docx via PyMuPDF — fully offline

pdf2docx parses PDF content streams with PyMuPDF and reconstructs them as structured DOCX elements. It runs entirely locally with no internet connection, no account, and no app launch.

Quality is good for text-heavy PDFs with simple layouts. Complex multi-column documents, PDFs generated from scans (rasterized text), or PDFs with unusual font encodings may not reconstruct accurately. This is a fundamental limitation of parsing PDF content streams, not a bug in pdf2docx.

The library is installed into an isolated venv at ~/.doc-converter/venv/ by the installer, so it never conflicts with system Python packages.

4. macOS Service registration

The critical missing piece: Info.plist

Bug Without Info.plist, pbs never discovers the workflow

An Automator .workflow bundle must contain Contents/Info.plist with a valid NSServices array. Without it, lsregister (Spotlight) returns -10811 and skips the bundle entirely. pbs -dump then shows no record of the service, and no context menu entry ever appears.

The fix: the installer now generates Info.plist alongside document.wflow, setting NSSendFileTypes to the correct UTIs for each conversion direction. 

Convert to PDF  → NSSendFileTypes:
  org.openxmlformats.wordprocessingml.document  (DOCX)
  com.microsoft.word.doc                         (legacy DOC)

Convert to Word → NSSendFileTypes:
  com.adobe.pdf                                  (PDF)

Services vs. Quick Actions

Decision Automator services appear under Services, not Quick Actions

In macOS 13+, the Finder right-click menu has two distinct sections:

  • Quick Actions — reserved for App Store extensions and system extensions (.appex bundles). Automator workflows cannot appear here.
  • Services — where user-created Automator workflows live. Accessible via right-click → Services → Convert to PDF / Convert to Word.

To auto-enable the services without requiring the user to visit System Settings → Keyboard → Shortcuts → Services, the installer writes enable flags directly into the pbs (pasteboard server) preference domain using a Python plistlib export/merge/import cycle. defaults -dict-add was ruled out because it cannot handle keys containing parentheses.

Why not killall cfprefsd?

Bug cfprefsd flush wipes unsaved preferences

Early versions of the installer ran killall cfprefsd after writing pbs preferences, following advice commonly found in Stack Overflow answers. This is incorrect: cfprefsd holds an in-memory write buffer for all preference domains. Killing it forces a flush that can lose any preferences the user modified since the last sync. The installer and uninstaller both avoid this. The services registry is refreshed instead with /System/Library/CoreServices/pbs -update.

5. User copy & message catalog

All user-facing strings live in MESSAGES at the top of convert.py. No inline English copy appears in conversion or dispatch logic. Error sentinels use a __sentinel__ prefix convention so dispatch_conversion can branch on them before they reach post_notification as raw text. 

Sentinel conventions:
  "__pkg_missing__"            → package_missing notification
  "__no_converter__"           → no_converter notification
  "__textutil_failed__<detail>" → textutil_failed notification
  "__chrome_failed__<detail>"  → chrome_failed notification

This keeps localization straightforward: to add a locale, duplicate MESSAGES under a new locale key and swap it at load time without touching conversion logic.

7. Open issues

  • Test coverage: convert.py has no automated tests. Unit tests for the conversion result dataclass, dispatch logic, and sentinel parsing would catch regressions without needing LibreOffice or Chrome installed in CI.
  • LibreOffice path is hardcoded to /Applications/LibreOffice.app/Contents/MacOS/soffice. A non-standard install location (e.g. /opt/homebrew/…) would silently fall through to the Chrome fallback. A which soffice probe before the hardcoded path check would handle this.
  • Batch error reporting shows only the first error message when multiple files fail. Each file's error is in the ConversionResult list and could be written to a temp log.
